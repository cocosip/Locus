# Locus 性能瓶颈与代码问题修复方案（2026-02-28）

> 目标：在不破坏现有 API 行为的前提下，先修复一致性问题（数据正确性），再降低高并发热点与维护任务耗时。

---

## 1. 问题优先级与修复顺序

### P0（必须先修）
1. **完成流程不一致：物理删除失败仍删元数据**，导致文件变成“孤儿”，后续可能被重建并重复入队。
2. **Windows 路径比较大小写问题**，孤儿检测可能误判并重复重建元数据。

### P1（并发热点）
3. **同租户分配锁竞争严重**：`GetNextPendingFileAsync` / `GetNextPendingBatchAsync` 全部串行在同一把 tenant lock。
4. **批量分配锁持有时间过长**：大量 stale/delayed key 时，锁内扫描过深。

### P2（维护任务性能）
5. **孤儿重建全量递归扫描**：大租户下 `AllDirectories` 扫描耗时、内存和 IO 压力高。

---

## 2. P0 修复方案

## 2.1 完成流程一致性（防重复入队）

### 现状问题
- `FileScheduler.MarkAsCompletedAsync` 在物理删除失败时记录错误但继续删元数据。
- `StoragePool.MarkAsCompletedAsync` 之后会减租户配额。
- 结果：文件仍在磁盘，但元数据已消失；后续 `CleanupOrphanedFilesAsync` 会重建为 Pending，形成重复处理风险。

### 修改目标
- **物理删除失败时，元数据不能被移除，也不能立即减配额。**
- 文件应进入“等待维护清理”的状态，而不是回到可处理队列。

### 推荐实现（低风险）
1. 在 `FileScheduler.MarkAsCompletedAsync` 中：
   - 仅当 `File.Delete` 成功或文件本就不存在（`FileNotFoundException` / `DirectoryNotFoundException`）时，才执行 `RemoveAsync`。
   - 对其他删除异常：
     - 克隆并更新 metadata：`Status = PermanentlyFailed`，`LastError` 写入 `COMPLETE_DELETE_FAILED` 前缀，`LastFailedAt = UtcNow`，清空 `ProcessingStartTime`。
     - `AddOrUpdateAsync(updated)` 持久化状态。
     - **抛出异常**，阻止上层配额递减。
2. `StoragePool.MarkAsCompletedAsync` 保持“仅在 scheduler 成功后减配额”的现有逻辑。
3. 维护任务（现有 `CleanupPermanentlyFailedFilesAsync`）继续回收这类文件并在回收时减目录配额。

### 需要新增/更新的测试
- 物理删除抛异常时：
  - metadata 不被删除；
  - `Status` 变为 `PermanentlyFailed` 且错误前缀正确；
  - `StoragePool.MarkAsCompletedAsync` 抛异常且不减租户配额。
- 文件不存在时（幂等完成）：仍可正常删 metadata 并减配额。

---

## 2.2 路径比较大小写一致性（Windows）

### 现状问题
- 孤儿检测 `knownPaths` 使用 `StringComparer.Ordinal`。
- Windows 路径大小写不敏感，可能把同一路径当成不同值，触发误判重建。

### 修改目标
- 路径比较策略与运行平台一致（Windows 忽略大小写，Linux 区分大小写）。

### 推荐实现
1. 在 `StorageCleanupService.CleanupOrphanedFilesAsync` 构建 `knownPaths` 时：
   - 先做 `Path.GetFullPath` 规范化。
   - 比较器使用：
     - Windows：`StringComparer.OrdinalIgnoreCase`
     - 其他平台：`StringComparer.Ordinal`
2. 对扫描出的 `physicalPath` 也做同样规范化后再 `Contains`。

### 需要新增/更新的测试
- 在 Windows 语义下，路径仅大小写不同不应被识别为孤儿。
- 在 Linux 语义下，大小写不同应视为不同路径（保持严格）。

## 3. P1 修复方案（并发热点）

## 3.1 先做“低侵入”优化（建议先落地）

### 修改目标
在不重构核心分配模型前，尽量缩短锁等待与锁持有时间。

### 推荐实现
1. **空队列快速返回（锁外）**
   - 在 `GetNextPendingFileAsync` / `GetNextPendingBatchAsync` 加：
     - 若 `_pendingFileCounts[tenantId] <= 0`，直接返回，不进入 tenant lock。
2. **动态收敛扫描上限**
   - `GetNextPendingBatchAsync` 的 `maxDequeues` 改为与 `batchSize`、`pendingCount` 相关的较小上限，例如：
     - `maxDequeues = Math.Min(pendingCount + 64, batchSize * 2 + 64)`。
   - 目标是避免 stale/delayed 多时长时间占锁。
3. **减少锁内重复时间调用**
   - 锁内 `DateTime.UtcNow` 只取一次并复用。

### 验收指标
- 100 并发 worker、同 tenant 场景下：
  - P95 分配耗时下降；
  - 锁等待时间下降；
  - 吞吐提升且无重复分配。

---

## 3.2 结构性优化（第二阶段，可选）

> 如果 3.1 后热点仍明显，再做此阶段。

### 方向
把“未到重试时间”的文件从主 pending 队列剥离，避免反复 dequeue/re-enqueue。

### 推荐实现
1. 每 tenant 维护两类结构：
   - `ReadyQueue`：立即可分配 key。
   - `DelayedQueue`：按 `AvailableForProcessingAt` 排序（最小堆/优先队列）。
2. 分配前先把到期 delayed 项提升到 ready，再从 ready 分配。
3. `MarkAsFailed`（带延迟）直接入 delayed，而不是先入 pending。

### 预期收益
- stale/delayed 造成的锁内扫描显著下降。
- 高失败重试场景下，分配抖动更小。

---

## 4. P2 修复方案（孤儿扫描性能）

## 4.1 先做“立刻见效”的优化

### 推荐实现
1. `GetFiles(..., SearchOption.AllDirectories)` 改为流式遍历（`EnumerateFiles`）。
2. 增加每轮扫描预算（例如 `MaxOrphanFilesPerRun`），超额后中断并记录续扫游标。
3. 先按 volume/tenant 分块处理，避免单次全量内存膨胀。

### 预期收益
- 峰值内存降低。
- 大目录下 GC 压力和启动抖动降低。

## 4.2 中期优化

1. 维护“最近写入目录”集合，优先扫描热目录。
2. 为每个 tenant+volume 记录上次扫描时间，支持增量扫描。

## 7. 实施顺序建议（下个会话可直接照此执行）

1. 先做 P0（正确性）并补齐单测。
2. 跑全量测试确认无回归。
3. 做 P1 低侵入优化（3.1），先验证收益。
4. 若收益不足，再做 P1 结构性优化（3.2）。
5. 最后做 P2 扫描优化并评估维护任务耗时。

> 建议每一步都保持“小步提交 + 可回滚”，不要把 P0/P1/P2 混成一个大改动。
